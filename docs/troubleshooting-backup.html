Troubleshooting Guide

Common issues, solutions, and diagnostic procedures for the Securaa platform

Troubleshooting Guide

This troubleshooting guide provides basic steps to help resolve common issues. Follow these general troubleshooting steps, and if problems persist, contact our support team for assistance.

General Troubleshooting Steps

Follow these general troubleshooting steps to resolve common issues. If problems persist after completing these steps, contact our support team.

Issue Severity Component Common Symptoms Quick Fix Time to Resolve
Installation Permission Errors High Installation Permission denied, sudo failures Check sudo access, fix ownership 5-15 min
Database Connection Issues Critical Database Connection refused, timeouts Check service status, restart DB 10-30 min
Network Connectivity Problems Medium Network Web UI unreachable, API timeouts Check firewall, verify ports 15-45 min
High Memory Usage Medium Performance System slow, OOM errors Restart services, check memory 10-30 min
SSL Certificate Issues High Security Browser warnings, cert errors Renew certificate, update config 20-60 min
API Authentication Failures High API 401/403 errors, invalid tokens Regenerate API keys, check perms 5-15 min
Integration Connection Failures Medium Integrations Failed data sync, timeout errors Check credentials, test connection 15-45 min
Log File Rotation Issues Low Maintenance Disk space full, large log files Enable logrotate, cleanup old logs 10-20 min
Backup Failures High Backup Backup jobs fail, corruption errors Check disk space, verify config 30-90 min
Service Startup Failures Critical System Services won't start, dependency errors Check dependencies, restart in order 15-45 min

Installation & Deployment Issues

Installation Fails with Permission Errors

High

Symptoms:

  • Installation scripts fail with "Permission denied" errors
  • Database initialization fails
  • Service startup errors due to insufficient privileges

Solution:

# Check current user permissions
whoami
id

# Verify sudo access
sudo -l

# Fix ownership issues
sudo chown -R securaa:securaa /opt/securaa
sudo chmod +x /opt/securaa/bin/*

# Verify installation directory permissions
ls -la /opt/securaa/
  1. Ensure the installation user has sudo privileges
  2. Verify the securaa user and group exist
  3. Check SELinux/AppArmor policies if applicable
  4. Run installation script with proper privileges

Database Connection Issues

Critical

Symptoms:

  • Services fail to start with database connection errors
  • "Connection refused" or "Connection timeout" messages
  • Authentication failures to database

Diagnostic Commands:

# Test database connectivity
telnet localhost 5432  # PostgreSQL
telnet localhost 3306  # MySQL

# Check database service status
sudo systemctl status postgresql
sudo systemctl status mysql

# Test database authentication
psql -h localhost -U securaa -d securaa_db
mysql -h localhost -u securaa -p securaa_db

# Check connection limits
psql -c "SELECT * FROM pg_stat_activity;" -U postgres
mysql -e "SHOW STATUS LIKE 'Threads_connected';" -u root -p

Solutions:

  1. Verify database service is running and listening on correct port
  2. Check database configuration in /opt/securaa/config/database.yml
  3. Ensure database user has appropriate permissions
  4. Verify firewall rules allow database connections
  5. Check for connection pool exhaustion

Network Connectivity Problems

Medium

Symptoms:

  • Web interface inaccessible
  • API calls timing out
  • Service discovery failures
  • Integration endpoints unreachable

Network Diagnostics:

# Check port bindings
sudo netstat -tlnp | grep securaa
sudo ss -tlnp | grep securaa

# Test connectivity
curl -I https://your-securaa-instance.com
wget --spider https://your-securaa-instance.com

# Check firewall rules
sudo iptables -L
sudo ufw status

# Verify DNS resolution
nslookup your-securaa-instance.com
dig your-securaa-instance.com

Service & Performance Issues

High Memory Usage

Medium

Symptoms:

  • System becomes unresponsive
  • Out of memory errors in logs
  • Services being killed by OOM killer
  • Slow query performance

Memory Diagnostics:

# Check memory usage
free -h
htop
ps aux --sort=-resident

# Check for memory leaks
valgrind --tool=memcheck --leak-check=yes /opt/securaa/bin/securaa-service

# Monitor memory over time
sar -r 1 10

# Check swap usage
swapon -s
cat /proc/swaps

Solutions:

  1. Increase system memory if consistently high usage
  2. Optimize database queries and indexing
  3. Adjust application memory limits in configuration
  4. Enable query caching to reduce memory overhead
  5. Schedule regular restarts if memory leaks persist

Disk Space Issues

High

Symptoms:

  • Services failing to write log files
  • Database write errors
  • "No space left on device" errors
  • Backup operations failing

Disk Space Diagnostics:

# Check disk usage
df -h
du -sh /opt/securaa/*
du -sh /var/log/securaa/*

# Find large files
find /opt/securaa -type f -size +100M -exec ls -lh {} \;
find /var/log -type f -size +100M -exec ls -lh {} \;

# Check inode usage
df -i

# Identify growing directories
ncdu /opt/securaa

Cleanup Solutions:

# Clean old log files
find /var/log/securaa -name "*.log" -mtime +30 -delete
find /var/log/securaa -name "*.log.*" -mtime +7 -delete

# Compress old logs
gzip /var/log/securaa/*.log.1

# Clean temporary files
rm -rf /tmp/securaa-*
rm -rf /opt/securaa/tmp/*

# Archive old data
tar -czf securaa-archive-$(date +%Y%m%d).tar.gz /opt/securaa/data/old/

Slow Performance & Timeouts

Medium

Symptoms:

  • Web interface loading slowly
  • API requests timing out
  • Database queries taking too long
  • Playbook execution delays

Performance Diagnostics:

# Check CPU usage
top
iostat 1 5
sar -u 1 10

# Monitor database performance
# PostgreSQL
psql -c "SELECT query, query_start, state FROM pg_stat_activity WHERE state = 'active';"
psql -c "SELECT schemaname,tablename,n_tup_ins,n_tup_upd,n_tup_del FROM pg_stat_user_tables;"

# MySQL
mysql -e "SHOW PROCESSLIST;" -u root -p
mysql -e "SHOW ENGINE INNODB STATUS;" -u root -p

# Check I/O performance
iotop
hdparm -t /dev/sda

Optimization Solutions:

  1. Add database indexes for slow queries
  2. Increase timeout values in configuration
  3. Enable query caching and connection pooling
  4. Scale horizontally with load balancing
  5. Optimize concurrent playbook execution limits

Integration & API Issues

Authentication Failures

High

Symptoms:

  • API calls returning 401 Unauthorized
  • Integration services failing to authenticate
  • OAuth token refresh failures
  • LDAP/AD authentication errors

Authentication Diagnostics:

# Test API key validity
curl -H "Authorization: Bearer YOUR_API_KEY" \
     https://your-securaa-instance.com/api/v1/auth/validate

# Check API key permissions
curl -H "Authorization: Bearer YOUR_API_KEY" \
     https://your-securaa-instance.com/api/v1/auth/permissions

# Test LDAP connectivity
ldapsearch -x -H ldap://your-ldap-server.com \
           -D "cn=admin,dc=company,dc=com" \
           -W -b "dc=company,dc=com"

# Verify SSL certificate
openssl s_client -connect your-securaa-instance.com:443

Solutions:

  1. Regenerate expired API keys
  2. Verify API key has required permissions
  3. Check OAuth client credentials
  4. Validate LDAP/AD server connectivity
  5. Update SSL certificates if expired

  5. Validate webhook endpoint SSL certificates

Rate Limiting Issues

Low

Symptoms:

  • API calls returning 429 Too Many Requests
  • Integration feeds getting throttled
  • Bulk operations failing
  • Automated scripts hitting limits

Rate Limit Diagnostics:

# Check current rate limit status
curl -I -H "Authorization: Bearer YOUR_API_KEY" \
     https://your-securaa-instance.com/api/v1/any-endpoint

# Response headers show rate limit info:
# X-RateLimit-Limit: 1000
# X-RateLimit-Remaining: 999
# X-RateLimit-Reset: 1640995200

# Monitor API usage
grep "429" /var/log/securaa/api.log | tail -20

Solutions:

  1. Implement exponential backoff in API clients
  2. Distribute requests across time periods
  3. Use bulk API endpoints where available
  4. Request rate limit increase for legitimate use cases
  5. Cache frequently accessed data

Emergency Procedures

Complete System Failure

Critical

Immediate Actions:

  1. Check system status and service health
  2. Review recent system logs for errors
  3. Attempt service restart procedures
  4. Escalate to emergency support if needed

Emergency Commands:

# Quick system check
sudo systemctl status securaa-*
docker ps -a  # If using containers

# Emergency restart
sudo systemctl restart securaa-all
# OR
sudo /opt/securaa/bin/emergency-restart.sh

# Collect diagnostics
sudo /opt/securaa/bin/collect-diagnostics.sh

# Emergency contact
echo "System failure at $(date)" | mail -s "URGENT: Securaa System Down" support@securaa.com

Database Corruption

High

Recovery Steps:

  1. Stop all Securaa services immediately
  2. Create emergency backup of current state
  3. Run database integrity checks
  4. Restore from latest good backup if needed
# Stop services
sudo systemctl stop securaa-*

# Backup current state
sudo tar -czf /backup/emergency-$(date +%Y%m%d-%H%M%S).tar.gz /opt/securaa/data

# PostgreSQL recovery
sudo -u postgres pg_dump securaa_db > /backup/emergency-db.sql
sudo -u postgres vacuumdb --analyze securaa_db

# MySQL recovery
mysqldump -u root -p securaa_db > /backup/emergency-db.sql
mysqlcheck -u root -p --auto-repair securaa_db

Security Incident

Medium

Incident Response:

  1. Isolate affected systems if compromise suspected
  2. Preserve logs and evidence
  3. Change all administrative passwords
  4. Review access logs for unauthorized activity
# Preserve evidence
sudo tar -czf /evidence/securaa-logs-$(date +%Y%m%d-%H%M%S).tar.gz /var/log/securaa/

# Check for unauthorized access
sudo last | grep securaa
sudo grep -i "failed\|unauthorized\|suspicious" /var/log/securaa/*

# Network isolation (if needed)
sudo iptables -A INPUT -j DROP
sudo iptables -A OUTPUT -j DROP

Diagnostic Tools & Scripts

System Health Check

Comprehensive system health assessment script

#!/bin/bash
# Securaa Health Check Script

echo "=== Securaa System Health Check ==="
echo "Timestamp: $(date)"

# Service Status
echo -e "\n=== Service Status ==="
for service in securaa-web securaa-api securaa-automation securaa-analytics; do
    status=$(systemctl is-active $service)
    echo "$service: $status"
done

# Resource Usage
echo -e "\n=== Resource Usage ==="
echo "Memory: $(free -h | awk '/^Mem:/ {print $3 "/" $2}')"
echo "Disk: $(df -h /opt/securaa | awk 'NR==2 {print $5 " used"}')"
echo "CPU: $(top -bn1 | grep "Cpu(s)" | awk '{print $2}' | cut -d'%' -f1)%"

# Database Connectivity
echo -e "\n=== Database Connectivity ==="
pg_isready -h localhost -p 5432 && echo "Database: OK" || echo "Database: ERROR"

# API Health
echo -e "\n=== API Health ==="
curl -s -o /dev/null -w "%{http_code}" http://localhost:8080/api/v1/health | \
    grep -q "200" && echo "API: OK" || echo "API: ERROR"

echo -e "\n=== Health Check Complete ==="

Log Collector

Automated log collection for support analysis

#!/bin/bash
# Log Collection Script

TIMESTAMP=$(date +%Y%m%d-%H%M%S)
OUTPUT_DIR="/tmp/securaa-logs-$TIMESTAMP"
mkdir -p "$OUTPUT_DIR"

echo "Collecting logs to $OUTPUT_DIR..."

# Application logs
cp -r /var/log/securaa "$OUTPUT_DIR/"

# System logs
journalctl -u securaa-* --since "24 hours ago" > "$OUTPUT_DIR/systemd.log"

# Configuration files (sanitized)
cp /opt/securaa/config/*.yml "$OUTPUT_DIR/"
sed -i 's/password:.*/password: [REDACTED]/' "$OUTPUT_DIR"/*.yml

# System information
uname -a > "$OUTPUT_DIR/system-info.txt"
lsb_release -a >> "$OUTPUT_DIR/system-info.txt"
df -h >> "$OUTPUT_DIR/system-info.txt"
free -h >> "$OUTPUT_DIR/system-info.txt"

# Create archive
tar -czf "securaa-logs-$TIMESTAMP.tar.gz" -C /tmp "securaa-logs-$TIMESTAMP"
echo "Logs collected: securaa-logs-$TIMESTAMP.tar.gz"

Network Connectivity Test

Test all network dependencies and integrations

#!/bin/bash
# Network Connectivity Test

echo "=== Network Connectivity Test ==="

# Test internal services
echo -e "\n--- Internal Services ---"
for port in 5432 8080 9200 6379; do
    timeout 5 bash -c "/dev/null && \
        echo "Port $port: OK" || echo "Port $port: FAILED"
done

# Test external dependencies
echo -e "\n--- External Dependencies ---"
external_hosts=(
    "8.8.8.8:53"
    "your-ldap-server.com:389"
    "threat-feed.example.com:443"
    "siem.company.com:514"
)

for host_port in "${external_hosts[@]}"; do
    host=$(echo $host_port | cut -d: -f1)
    port=$(echo $host_port | cut -d: -f2)
    timeout 5 bash -c "/dev/null && \
        echo "$host:$port: OK" || echo "$host:$port: FAILED"
done

# DNS resolution test
echo -e "\n--- DNS Resolution ---"
for domain in "your-securaa-instance.com" "google.com"; do
    nslookup $domain >/dev/null 2>&1 && \
        echo "$domain: OK" || echo "$domain: FAILED"
done
